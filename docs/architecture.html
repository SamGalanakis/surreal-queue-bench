§ 01 — OverviewThe shape of the approach

Composite record-IDs encode queue order so reads are primary-key range scans. Buckets shard work; client-side HRW over the live worker set shards ownership; lease tokens give at-least-once delivery.

I · ORDER

Ordering lives in the record-ID

job_ready IDs are [queue, bucket, available_ns, job_key]. "Give me ready jobs" is a RecordIdScan range read — never a secondary-index seek.

II · SHARDS

Buckets shard work; HRW shards ownership

Producers hash into buckets, ignorant of consumers. Every consumer reads the live worker set and computes HRW locally — no persisted partition lease, no coordinator.

III · LEASES

TTL by ID — no timer service

job_lease IDs encode lease_until_ns. Recovery is itself a range scan up to now_ns. No per-row TTL, no scheduler.

§ 02 — Data flowStates & transitions

FIG. 2 — Token state machine 
stateDiagram-v2

Four states, four transactions. Ready and Leased are materialised as separate skinny tables — each transition is a delete-from-one, insert-into-other. The terminal states (Done, Dead) live only on the job_payload row.

FIG. 3 — What each transaction does
ENQUEUE
producer · insert_jobs
  1. BEGIN TRANSACTIONone TX per batch
  2. INSERT job_payloadstate = active
  3. INSERT job_readyid = [Q, B, 0, key]
  4. COMMIT TRANSACTIONpayload + token together
CLAIM
fn::bench_receive_ready
  1. DELETE (SELECT VALUE id FROM job_ready range) RETURN BEFORE[Q, B, NONE, NONE] ..= [Q, B, MAX, ~~~~]
  2. INSERT job_lease for eachid = [Q, B, lease_until_ns, key] · attempts+1
  3. RETURN [{lease_token, sent_at_ns}]
ACK
fn::bench_ack
  1. DELETE $lease_tokens RETURN BEFORE
  2. UPDATE leases[].jobstate = done · completed_at_ns
  3. RETURN count
RECOVER
fn::bench_recover_buckets
  1. FOR each owned bucket B
  2. DELETE (SELECT VALUE id FROM job_lease range) RETURN BEFORE[Q, B, NONE, NONE] ..= [Q, B, now_ns, ~~~~]
  3. attempts < max → INSERT job_ready
  4. attempts ≥ max → UPDATE state = dead

A fourth function, fn::bench_fail, is defined for explicit nack (delete lease → requeue or dead-letter), mirroring RECOVER's branching. The current benchmark loop never fails a job, so it's a defined capability rather than a hot path.

Concurrent mode runs the producer and consumer paths in parallel against the same SurrealDB; drain-only completes every enqueue first and only then starts the consumer side; insert-only stops after the enqueue half.

§ 03 — ExecutionWorkers, modes, ownership

INSERT-ONLY

N producers, no consumers.

→ pure enqueue cost

CONCURRENT

Producers + consumers in parallel.

→ steady-state throughput & e2e latency

DRAIN-ONLY

All inserts finish, then consumers drain.

→ fixed work-down · sdbjq-comparable

Passing --num-total-jobs N implies drain-only: the run inserts exactly N jobs before consumers start, so the drain rate is comparable across cells. --num-total-jobs with --mode insert-only is rejected; it would be the same as a plain insert-only run.

FIG. 5 — Bucket ownership via HRW (rendezvous hashing)

For every (bucket, worker) pair the score is xxh3(bucket_le_bytes ‖ worker_id). The worker with the highest score owns that bucket. No coordinator, no leader election — every live worker independently agrees, recomputing from the rows of worker whose lease_until > now.

When a worker dies, its worker.lease_until falls into the past and it drops out of the active set. Every survivor's next refresh recomputes HRW against the smaller set and naturally inherits the orphaned buckets, then calls fn::bench_recover_buckets to sweep their expired leases.

§ 04 — AbstractionsThe ID layout that makes it work

Every queue operation is a RecordIdScan range read or a delete-by-ID. There is no secondary index. The structure of the record-IDs is the access plan.

FIG. 7 — Record-ID anatomy
job_payload : job_key
job_ready : queuebucketavailable_nsjob_key
job_lease : queuebucketlease_until_nsjob_key
worker : worker_id
time slot in the ID stable identity of the underlying job worker UUID (single-segment ID)

available_ns reserves a structural slot for scheduled delivery — the receive call passes an upper bound, so a future delivery time would simply not match the range. The benchmark always writes available_ns = 0 and queries with MAX_INT; the field exists so the same access pattern extends to delayed jobs without a schema change. lease_until_ns, in contrast, is updated on every claim and is genuinely the moving sort key.

I · RANGE TOKENS

IDs encode the sort order

"Next ready job" = SELECT FROM job_ready:[Q,B,NONE,NONE]..=[Q,B,now,~~~] LIMIT N. Primary-key scan. Cheapest read the engine has.

II · VIRTUAL BUCKETS

Sharding lives only in the ID

No physical partitions. bucket = xxh3(job_key) % B. Default 1024; keep buckets ≫ consumers.

III · HRW OWNERSHIP

Client-side, no coordinator

Every worker reads the live worker rows and independently picks the same winner per bucket. The only persisted lease is the worker row itself.

IV · TTL BY ID

Lease tokens carry their deadline

lease_until_ns sits inside the ID. Recovery is a range scan up to now_ns. No per-row TTL, no scheduler.

V · ATOMIC TRANSITIONS

Stored functions in transactions

Each state change is one round-trip. Multi-row mutations commit together or roll back together.

VI · OBSERVABILITY

Atomics + sample buffers per op

Six op-kinds (producer, select, ack, register, heartbeat, refresh). p50/p95/p99 + captured EXPLAIN ANALYZE.

fn::bench_receive_ready — the range-scan claim 
DEFINE FUNCTION OVERWRITE fn::bench_receive_ready(
    $queue: string, $bucket: int, $limit: int,
    $available_until_ns: int, $lease_until_ns: int
) -> array<object> {
    // Atomic claim — one statement: DELETE wraps a bounded RecordIdScan.
    // Range tokens are deleted and the prior rows are returned for lease minting.
    LET $ready = DELETE (
        SELECT VALUE id
        FROM job_ready:[$queue, $bucket, NONE, NONE]
              ..=[$queue, $bucket, $available_until_ns, "~~~~"]
        LIMIT $limit
    ) RETURN BEFORE;
    IF $ready.len() = 0 { RETURN []; };

    // Mint skinny lease tokens. The ID encodes the deadline, so recovery is
    // itself a range scan — no per-row TTL, no timer service.
    INSERT INTO job_lease $ready.map(|$row| {
        id: type::record("job_lease", [$queue, $bucket, $lease_until_ns, $row.job_key]),
        queue: $queue, bucket: $bucket, lease_until_ns: $lease_until_ns,
        job: $row.job, job_key: $row.job_key,
        attempts: $row.attempts + 1, max_attempts: $row.max_attempts,
        sent_at_ns: $row.sent_at_ns
    }) RETURN NONE;

    RETURN $ready.map(|$row| {
        lease_token: type::record("job_lease",
            [$queue, $bucket, $lease_until_ns, $row.job_key]),
        sent_at_ns: $row.sent_at_ns
    });
};

§ 05 — ModulesWhat lives where

Two source files, two concerns: the benchmark itself, and the report it leaves behind. Everything else — SurrealQL schema, stored functions, JSON shape — is embedded as string constants inside main.rs.

CORE · 1,984 LOC

src/main.rs

Single-binary entry point. Parses CLI args with clap, connects to SurrealDB over WebSocket, defines the schema + stored functions on each run, and drives the producer/consumer Tokio tasks. Emits a structured JsonReport (or MatrixReport in matrix mode).

  • Args + BenchMode — CLI surface, matrix knobs
  • setup_schema — inline SurrealQL: tables + 4 stored fns
  • run_concurrent_benchmark / run_insert_only / run_fixed_workdown
  • run_producer + insert_jobs — batched BEGIN/COMMIT enqueue
  • run_consumer — per-bucket round-robin claim loop with empty-bucket backoff
  • partition_lease_targets — client-side HRW from live workers
  • refresh_buckets / recover_expired_leases — sweep stale leases
  • capture_explain — EXPLAIN ANALYZE for receive + recover ranges
  • BenchCounters — atomic op tallies + sample buffers for percentiles
src/main.rs · 1 module · tokio main
REPORT · 3,357 LOC

src/html_report.rs

Self-contained HTML report generator. Given a JsonReport or MatrixReport, produces a single static .html file with embedded charts (throughput, phase windows, latency CDFs, op percentiles, sparklines, query plans). No JS frameworks, no external CSS — everything inlined so the artifact is portable.

  • write_html_report — entry point invoked from main
  • html_output_path — infers .html path from --run, --output, or --export
  • Inline SVG charts, config tables, counters, and EXPLAIN dumps
  • Matrix mode: aggregate dashboard across every cell
